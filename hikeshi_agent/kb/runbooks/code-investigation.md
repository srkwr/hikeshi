# Runbook: コード起因の調査（diff の読み方）と code_fix の進め方

（合成サンプル KB・実データではない）

## いつ使うか
- 直近デプロイの後に異常が出ているが、**ロールバックで直せない可能性**（書込済みデータへの影響）があるとき。
- 変更差分（diff）から原因コードを特定したいとき。

## diff の読み方（疑わしい変更の優先順）
1. **書込経路の変更**（DB 書込・課金・在庫・ポイント等の更新ロジック）＝最優先で確認。
   ここに不具合があると**不可逆**（巻き戻してもデータは壊れたまま）になりやすい。
2. **リトライ/タイムアウト/例外処理の変更**＝部分的成功や多重実行を生み、書込の重複・欠落につながる。
3. **読み取り専用の変更**（表示・整形・ログ）＝可逆。ロールバックで安全に戻せることが多い。
4. 依存ライブラリの更新・設定値の変更は、コードでなく config/依存side の Runbook を参照。

## 対処の決め方（安全境界と接続）
- 可逆（読み取り系の不具合・データ破損なし）→ **rollback**（自動許容の範囲）。
- **不可逆（書込済みデータに影響・例: 過剰請求・データ破損）→ rollback では直らない。
  `code_fix`（修正の提案）＋データ補正手順を提示し、必ず人間承認（HITL）**。
  巻き戻しは新たな書込を止めるだけで、壊れたデータは残ることに注意。

## code_fix の進め方（提案のみ・実行しない）
1. 原因 diff の特定（該当リビジョン・該当ファイル/関数を明示）。
2. 修正方針の提案（最小修正・テスト追加・再発防止）。
3. 影響データの範囲特定と補正手順（バックフィル/再計算）を別タスクとして提示。
4. 人間が承認してから修正デプロイ。**エージェントはコードを直接変更しない**。
